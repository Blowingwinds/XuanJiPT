# 第四部分开发文档：面向 PTT-DAG 的原子算子模板库与动作副作用建模

## 1. 本部分定位

本部分对应一阶段工作的第四项：原子算子模板库设计。

第二部分定义了 `PTT-DAG 四级任务规划模型`，第三部分定义了 `基于 ATT&CK 战术路线模板的任务解耦机制`。第四部分要解决的是：

```text
Technique 如何落到标准化 Action？
Action 的来源如何被约束？
Action 如何携带输入、输出、风险、副作用和工具映射信息？
```

因此，本部分的核心是建立一个标准化原子算子模板库，使系统生成的每个 Action 都能追溯到已登记的 Operator Template。

## 2. 核心思想

### 2.1 Technique、Operator、Action 的关系

本研究中需要区分三类对象：

```text
Technique：技术方法，来自 ATT&CK、本地 SOP 或路线模板。
Operator Template：原子算子模板，是标准化动作定义。
Action Instance：任务图中的具体动作实例，由算子模板绑定目标后生成。
```

关系如下：

```text
Technique -> Operator Template -> Action Instance
```

示例：

```text
service_discovery
  -> op_service_fingerprint
  -> act-service-fingerprint-001
```

其中：

- `service_discovery` 表示“服务识别”这一技术方法；
- `op_service_fingerprint` 表示“服务指纹识别”算子模板；
- `act-service-fingerprint-001` 表示“对 192.168.56.10:80 执行服务指纹识别”这一具体 Action。

### 2.2 为什么需要原子算子模板库

原子算子模板库具有四个作用：

1. 约束 LLM 输出
   - LLM 不能生成任意动作，只能从已登记算子中选择或由程序实例化。

2. 统一动作语义
   - 不同 Technique 最终都落到统一的 Action 描述结构。

3. 支撑工具适配
   - 第五部分的工具 Adapter 根据 `operator_id` 判断是否支持某类 Action。

4. 支撑风险控制
   - 每个算子预先定义风险等级、副作用和是否需要人工审批。

## 3. 原子算子定义

原子算子是指可以被工具执行或模拟执行的最小动作模板，具备明确输入、输出、前置条件、风险等级、副作用描述和候选工具集合。

原子算子不是具体执行记录，而是可复用的动作定义。

例如：

```text
op_service_fingerprint 是模板；
对 192.168.56.10:80 执行服务指纹识别 是模板实例化后的 Action。
```

## 4. 算子模板 Schema

建议一阶段使用 YAML 文件维护算子模板库，例如：

```text
data/operator_templates.yaml
```

单个算子模板建议采用以下结构：

```yaml
operator_id: op_service_fingerprint
name: service_fingerprint
display_name: 服务指纹识别
description: 识别目标端口对应的服务名称和版本信息
category: discovery
mapped_techniques:
  - service_discovery
required_inputs:
  - host
  - port
optional_inputs:
  - protocol
  - timeout
outputs:
  - service_name
  - version
  - banner
  - confidence
preconditions:
  - target_in_scope
  - port_open
risk_level: low
side_effects:
  network_noise: low
  service_disruption: none
  credential_exposure: none
  state_change: none
  audit_log_impact: low
requires_human_approval: false
allowed_tools:
  - nmap_adapter
  - web_probe_adapter
result_schema:
  service_name: string
  version: string
  banner: string
  confidence: number
```

字段说明：

| 字段 | 含义 |
|---|---|
| operator_id | 算子唯一标识，Action 必须绑定该字段 |
| name | 机器可读名称 |
| display_name | 人类可读名称 |
| description | 算子功能说明 |
| category | 算子类别 |
| mapped_techniques | 可映射到该算子的 Technique 列表 |
| required_inputs | 实例化 Action 时必须具备的输入 |
| optional_inputs | 可选输入 |
| outputs | 算子执行后预期产生的数据字段 |
| preconditions | 执行前置条件 |
| risk_level | 风险等级 |
| side_effects | 动作副作用描述 |
| requires_human_approval | 是否需要人工审批 |
| allowed_tools | 可执行该算子的工具 Adapter |
| result_schema | 标准化结果字段类型 |

## 5. 动作副作用建模

### 5.1 副作用维度

为了支撑安全策略、后续目标黑板和多智能体冲突仲裁，本研究为每个算子设计副作用描述。

建议一阶段采用离散等级建模，不做复杂数学计算。

```yaml
side_effects:
  network_noise: none | low | medium | high
  service_disruption: none | low | medium | high
  credential_exposure: none | low | medium | high
  state_change: none | temporary | persistent
  audit_log_impact: none | low | medium | high
```

字段解释：

| 维度 | 含义 |
|---|---|
| network_noise | 是否产生明显网络流量 |
| service_disruption | 是否可能影响服务稳定性 |
| credential_exposure | 是否涉及凭据尝试或敏感数据 |
| state_change | 是否改变目标系统状态 |
| audit_log_impact | 是否容易产生审计日志 |

### 5.2 风险等级规则

一阶段可采用人工标注为主、规则推导为辅的方式。

建议规则：

```text
如果 state_change = persistent，则 risk_level 至少为 high；
如果 credential_exposure = medium 或 high，则 risk_level 至少为 high；
如果 service_disruption = medium 或 high，则 risk_level 至少为 medium；
如果 network_noise = high，则 risk_level 至少为 medium；
如果所有副作用均为 none 或 low，则 risk_level 可为 low。
```

### 5.3 人工审批规则

```text
risk_level = low：默认可自动执行；
risk_level = medium：授权范围内可自动执行；
risk_level = high：默认 waiting_approval；
risk_level = critical：默认只规划，不自动执行。
```

对应字段：

```yaml
requires_human_approval: true
```

高风险算子必须设置该字段为 true。

## 6. 首批算子清单

一阶段建议设计 12 个算子，其中 10 个低风险或中低风险算子用于默认路线，2 个高风险算子作为审批机制占位。

| 算子 ID | 名称 | 风险等级 | 一阶段处理方式 |
|---|---|---|---|
| op_host_discovery | 主机存活探测 | low | 可真实或模拟 |
| op_port_scan | 端口状态识别 | medium | 可真实或模拟 |
| op_service_fingerprint | 服务指纹识别 | low | 可真实或模拟 |
| op_web_title | Web 标题识别 | low | 可真实或模拟 |
| op_web_header_analysis | Web 响应头分析 | low | 可真实或模拟 |
| op_web_tech_stack | Web 技术栈识别 | low | 可真实或模拟 |
| op_tls_certificate | TLS 证书信息提取 | low | 可真实或模拟 |
| op_cve_lookup | 已知漏洞库匹配 | low | 本地数据匹配 |
| op_config_risk_check | 配置风险检查 | low | 模拟或规则判断 |
| op_report_summary | 报告汇总 | low | 本地生成 |
| op_weak_credential_check | 弱凭据风险检查 | high | 仅模板，不自动执行 |
| op_default_credential_check | 默认凭据检查 | high | 仅模板，不自动执行 |

## 7. Technique 到 Operator 映射

第三部分已经提出 Technique 到 Operator 的映射关系，第四部分需要进一步明确这些 operator_id 必须真实存在于算子模板库。

建议映射如下：

```yaml
host_discovery:
  candidate_operators:
    - op_host_discovery

port_discovery:
  candidate_operators:
    - op_port_scan

service_discovery:
  candidate_operators:
    - op_service_fingerprint

web_information_discovery:
  candidate_operators:
    - op_web_title
    - op_web_header_analysis
    - op_web_tech_stack

web_title_detection:
  candidate_operators:
    - op_web_title

web_header_analysis:
  candidate_operators:
    - op_web_header_analysis

web_tech_stack_identification:
  candidate_operators:
    - op_web_tech_stack

tls_certificate_inspection:
  candidate_operators:
    - op_tls_certificate

cve_lookup:
  candidate_operators:
    - op_cve_lookup

config_risk_check:
  candidate_operators:
    - op_config_risk_check

weak_credential_check:
  candidate_operators:
    - op_weak_credential_check

default_credential_check:
  candidate_operators:
    - op_default_credential_check

report_summary:
  candidate_operators:
    - op_report_summary
```

约束规则：

```text
Technique 必须至少映射到一个 Operator；
Operator 必须存在于 operator_templates.yaml；
Action 只能由 Operator 实例化生成；
没有合法 Operator 的 Technique 不允许进入最终 PTT-DAG。
```

## 8. Operator 到 Action 的实例化规则

### 8.1 实例化流程

```text
1. 输入 selected Technique；
2. 查询 Technique-Operator 映射表；
3. 获取候选 Operator Template；
4. 检查 required_inputs 是否满足；
5. 绑定目标资产、端口、服务等上下文；
6. 生成 Action Instance；
7. 从 Operator Template 继承风险、副作用、输出字段和 allowed_tools；
8. 写入 PTT-DAG Action 节点。
```

### 8.2 示例

算子模板：

```yaml
operator_id: op_port_scan
required_inputs:
  - host
outputs:
  - open_ports
risk_level: medium
allowed_tools:
  - nmap_adapter
  - mock_adapter
```

实例化后的 Action：

```json
{
  "node_id": "act-port-scan-001",
  "level": "Action",
  "name": "port_scan",
  "parent_id": "tech-port-discovery-001",
  "operator_id": "op_port_scan",
  "target": {
    "asset_id": "asset-001",
    "host": "192.168.56.10"
  },
  "inputs": {
    "host": "192.168.56.10",
    "scan_profile": "top_ports"
  },
  "outputs": ["open_ports"],
  "risk_level": "medium",
  "side_effects": {
    "network_noise": "medium",
    "service_disruption": "low",
    "credential_exposure": "none",
    "state_change": "none",
    "audit_log_impact": "medium"
  },
  "allowed_tools": ["nmap_adapter", "mock_adapter"],
  "depends_on": ["act-host-discovery-001"],
  "status": "pending",
  "result": null
}
```

## 10. 算子模板校验规则

算子模板库加载后应进行静态校验。

校验规则：

```text
1. operator_id 必须唯一；
2. name 不为空；
3. mapped_techniques 至少包含一个 Technique；
4. required_inputs 必须为数组；
5. outputs 必须为数组；
6. risk_level 必须属于 low / medium / high / critical；
7. side_effects 必须包含规定维度；
8. high / critical 算子必须 requires_human_approval = true；
9. allowed_tools 至少包含一个工具 Adapter；
10. result_schema 必须覆盖 outputs 中的关键字段。
```

## 11. 与第三部分的关系

第三部分负责：

```text
选择路线 -> 选择 Technique -> 查找 candidate operator_id
```

第四部分负责：

```text
定义 operator_id 的完整模板 -> 支撑 Action 实例化 -> 提供风险和副作用信息
```

因此，第三部分输出的 operator_id 必须在第四部分的模板库中存在。

## 12. 与第五部分的关系

第五部分工具语义封装与自适应调用模块会使用第四部分中的以下字段：

```text
operator_id
required_inputs
outputs
risk_level
side_effects
allowed_tools
result_schema
```

工具 Adapter 是否能执行某个 Action，首先取决于它是否支持该 Action 的 `operator_id`。

## 14. 最低验收标准

本部分至少满足：

- 能说明原子算子是什么；
- 能区分 Technique、Operator Template 和 Action Instance；
- 能定义算子模板字段；
- 能定义动作副作用维度；
- 能给出首批至少 10 个算子；
- 能说明算子如何实例化为 Action；
- 能说明高风险算子为何不自动执行；
- 能支撑第五部分工具 Adapter 根据 operator_id 匹配能力。

## 15. 可写入论文的表述

可在论文中表述为：

> 为避免大语言模型在渗透测试任务规划过程中生成不可控或不可执行的动作，本文设计了面向 PTT-DAG 的原子算子模板库。原子算子作为 Technique 与 Action 之间的标准化中间层，定义了每类可执行动作的输入、输出、前置条件、风险等级、副作用和候选工具集合。任务解耦模块生成 Action 时必须绑定合法 operator_id，且该 operator_id 必须存在于算子模板库中。通过这种模板化约束，系统将自由生成式规划转化为基于专家知识和标准化动作模板的受控规划过程。同时，本文为每个算子引入网络噪声、服务扰动、凭据暴露、状态改变和审计日志影响等副作用维度，为后续工具自适应调用、目标黑板状态回填以及多智能体冲突仲裁提供基础。
