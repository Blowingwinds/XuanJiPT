# 一阶段开发指导：论文导向的 PTT-DAG 任务规划原型

## 1. 阶段定位

本项目当前阶段不是以完整工程落地为目标，而是服务于毕业论文中期成果建设。当前工作应围绕开题报告截至 2026 年 6 月底应完成的内容，形成“理论模型 + 机制设计 + 最小原型框架”的阶段性成果。

截至 2026 年 6 月 30 日，按开题报告进度，应补齐：

- 2026.03 - 2026.04：基于 ATT&CK 专家知识约束的任务解耦机制，构建 Goal-Tactic-Technique-Action 四级 PTT 任务树模型。
- 2026.05 - 2026.06：异构渗透工具能力语义封装技术，建立基于 RAG/知识检索的原子算子模板库及动作副作用评估模型，完成渗透工具语义适配与自适应调用模块设计。

经过讨论，一阶段方案统一调整为：

```text
PTT-DAG 四级任务规划模型
  + ATT&CK 固定战术路线模板约束
  + LLM 受限 Technique 选择
  + 原子算子模板库
  + 工具语义封装与自适应调用
```

## 2. 一阶段总体目标

一阶段目标不是实现完整自动化渗透测试系统，而是形成一个可支撑论文论证和后续原型开发的任务规划框架。

核心闭环为：

```text
用户目标 / 环境画像
  -> 固定 ATT&CK 战术路线模板匹配
  -> LLM 在候选 Technique 中受限选择
  -> Technique 映射 Operator Template
  -> Operator 实例化 Action
  -> 生成 PTT-DAG
  -> 后续工具 Adapter 自适应调用
```

该闭环的重点是可解释、可追溯、可校验，而不是高风险实战执行。

## 3. 一阶段五个组成部分

一阶段包含五个部分：

1. 总体技术架构设计
2. PTT-DAG 四级任务规划模型
3. 基于 ATT&CK 战术路线模板的任务解耦机制
4. 面向 PTT-DAG 的原子算子模板库与动作副作用建模
5. 工具语义封装与自适应调用模块设计

其中前四部分已经明确设计方向，第五部分接下来继续讨论并落文档。

## 4. 第一部分：总体技术架构设计

### 4.1 目标

总体架构用于说明系统如何从用户目标逐步生成受约束的任务规划，并为后续多智能体协同和目标黑板机制预留接口。

### 4.2 推荐分层

```text
┌──────────────────────────────┐
│ 1. 输入与环境画像层           │
│ 用户目标、授权范围、资产信息   │
└──────────────┬───────────────┘
               ↓
┌──────────────────────────────┐
│ 2. 知识约束层                 │
│ ATT&CK 路线模板、本地 SOP、算子库 │
└──────────────┬───────────────┘
               ↓
┌──────────────────────────────┐
│ 3. 任务解耦与规划层           │
│ 路线匹配、Technique 选择、PTT-DAG │
└──────────────┬───────────────┘
               ↓
┌──────────────────────────────┐
│ 4. 动作模板层                 │
│ Operator Template、风险、副作用 │
└──────────────┬───────────────┘
               ↓
┌──────────────────────────────┐
│ 5. 工具语义适配层             │
│ Adapter、工具能力注册、自适应选择 │
└──────────────┬───────────────┘
               ↓
┌──────────────────────────────┐
│ 6. 状态反馈与报告层           │
│ Action 结果、状态回填、报告输出 │
└──────────────────────────────┘
```

### 4.3 论文表述重点

- 不让 LLM 直接生成工具命令；
- 不让 LLM 自由生成完整攻击链；
- 通过固定战术路线模板收敛规划空间；
- 通过 Technique 候选集合限制 LLM 选择范围；
- 通过原子算子模板库约束 Action 来源；
- 通过 Adapter 层屏蔽异构工具差异。

## 5. 第二部分：PTT-DAG 四级任务规划模型

### 5.1 设计结论

第二部分不再采用单纯树模型，而是采用：

```text
PTT 语义层级 + Action 执行 DAG
```

即：

```text
Goal / Tactic / Technique：负责语义分解和知识归类；
Action DAG：负责执行依赖、数据流和并行调度。
```

### 5.2 四级定义

| 层级 | 含义 | 示例 |
|---|---|---|
| Goal | 总体测试目标 | 对授权靶场主机进行基础安全评估 |
| Tactic | 战术阶段 | 侦察、发现、漏洞线索分析、报告 |
| Technique | 技术方法 | 主机发现、端口发现、服务识别、CVE 匹配 |
| Action | 原子动作实例 | 对 192.168.56.10 执行端口状态识别 |

### 5.3 关系类型

PTT-DAG 包含两类核心关系：

```text
contains：Goal -> Tactic -> Technique -> Action 的语义包含关系；
depends_on：Action -> Action 的执行依赖关系。
```

示例：

```text
Goal: 基础安全评估
  Tactic: 侦察
    Technique: 主机发现
      Action A: 主机存活探测
    Technique: 端口发现
      Action B: 端口扫描
    Technique: 服务识别
      Action C: 服务指纹识别

Action DAG:
A -> B -> C
```

### 5.4 LangGraph 契合点

PTT-DAG 可自然映射到 LangGraph：

- Planner Agent：生成 PTT-DAG；
- Validator Agent：校验范围、风险、依赖和模板合法性；
- Scheduler Agent：根据 DAG 选择 ready Action；
- Tool Agent：调用工具 Adapter；
- Parser Agent：解析工具输出；
- Blackboard Updater：回填状态；
- Reporter Agent：生成报告。

## 6. 第三部分：ATT&CK 战术路线模板约束任务解耦机制

### 6.1 设计结论

第三部分采用：

```text
固定 ATT&CK 战术路线模板 + LLM 受限 Technique 选择 + 程序校验
```

LLM 的角色是受限选择器，不是自由规划器。

### 6.2 默认路线

一阶段定义 3 条默认启用路线和 1 条默认禁用高风险路线。

| 路线 ID | 名称 | 状态 |
|---|---|---|
| route_basic_exposure_assessment | 基础暴露面评估路线 | 默认启用 |
| route_web_service_risk_assessment | Web 服务风险评估路线 | 默认启用 |
| route_internal_asset_discovery | 内网资产发现路线 | 默认启用 |
| route_credential_risk_assessment | 凭据风险检查路线 | 默认禁用 |

### 6.3 解耦流程

```text
Step 1: 目标解析
Step 2: 环境画像归一化
Step 3: 战术路线模板匹配
Step 4: LLM 在候选 Technique 中受限选择
Step 5: Technique 映射 Operator
Step 6: Operator 实例化 Action
Step 7: 合法性校验并输出 PTT-DAG
```

### 6.4 关键约束

- LLM 只能从候选 Technique 中选择；
- LLM 不允许新增路线；
- LLM 不允许新增 operator_id；
- 不允许输出真实攻击命令；
- 高风险动作必须进入 `waiting_approval`；
- 最终结果必须通过程序校验。

## 7. 第四部分：原子算子模板库与动作副作用建模

### 7.1 设计结论

第四部分建立 `Operator Template` 作为 Technique 与 Action 之间的标准化中间层。

```text
Technique -> Operator Template -> Action Instance
```

示例：

```text
service_discovery
  -> op_service_fingerprint
  -> act-service-fingerprint-001
```

### 7.2 算子模板核心字段

```yaml
operator_id: op_service_fingerprint
name: service_fingerprint
display_name: 服务指纹识别
mapped_techniques:
  - service_discovery
required_inputs:
  - host
  - port
outputs:
  - service_name
  - version
  - banner
risk_level: low
side_effects:
  network_noise: low
  service_disruption: none
  credential_exposure: none
  state_change: none
  audit_log_impact: low
requires_human_approval: false
allowed_tools:
  - nmap_adapter
  - web_probe_adapter
result_schema:
  service_name: string
  version: string
  banner: string
```

### 7.3 副作用维度

```text
network_noise：网络噪声；
service_disruption：服务扰动；
credential_exposure：凭据暴露；
state_change：目标状态改变；
audit_log_impact：审计日志影响。
```

### 7.4 首批算子

一阶段采用 12 个算子：

| 算子 ID | 名称 | 风险 |
|---|---|---|
| op_host_discovery | 主机存活探测 | low |
| op_port_scan | 端口状态识别 | medium |
| op_service_fingerprint | 服务指纹识别 | low |
| op_web_title | Web 标题识别 | low |
| op_web_header_analysis | Web 响应头分析 | low |
| op_web_tech_stack | Web 技术栈识别 | low |
| op_tls_certificate | TLS 证书信息提取 | low |
| op_cve_lookup | 已知漏洞库匹配 | low |
| op_config_risk_check | 配置风险检查 | low |
| op_report_summary | 报告汇总 | low |
| op_weak_credential_check | 弱凭据风险检查 | high |
| op_default_credential_check | 默认凭据检查 | high |

高风险算子仅用于论文说明和审批机制占位，不自动执行。
